# Get 时延敏感场景专项分析（5ms / 20ms）

## 目标

评估时延敏感 Get 请求在超短超时预算（`5ms`、`20ms`）下的真实重试行为、超时返回特征以及 timeout 后潜在残余流量（“垃圾流量”）风险。

---

## 1. 关键实现前提

基于当前代码实现（`RetryOnError` / `RetryOnErrorRepent` 与 Get 主链路）：

1. **重试继续条件存在硬门槛**  
   `RetryOnError` 内部要求剩余时间满足最小单次执行门槛：
   - `minOnceRpcTimeoutMs = 50ms`
   - 若 `remainTimeMs < 50ms`，不会继续进入下一轮重试。

2. **首轮调用仍会执行**  
   即使总超时只有 5ms/20ms，首轮 RPC 仍会发起（单次 rpc timeout 至少会被设置为 1ms）。

3. **Worker 远端拉取额外刹车**  
   在 `TryGetObjectFromRemote`，若 `remainTimeMs <= 100ms`，会停止继续 retry，并将待重试对象转入失败集合返回。

4. **超时预算为全链路共享递减**  
   `reqTimeoutDuration.CalcRealRemainingTime()` 在 worker 内多层调用共享，master 查询、remote worker 拉取都会消耗同一预算。

---

## 2. timeout = 5ms 行为分析

## 2.1 重试次数

- 结论：**几乎等价于“无重试”**。
- 常态表现：**1 次尝试后直接结束**（0 次有效 retry）。

原因：

- 首轮执行后剩余时间几乎必然小于 50ms，不满足继续重试条件。

## 2.2 成功路径特征

- 只适用于极高概率本地命中（内存/本地快速路径）。
- 涉及 `worker -> master` 或 `worker -> remote worker` 的路径几乎没有可用时间窗。

## 2.3 timeout 返回与残余流量

- Get 请求可正常返回 timeout（`K_RPC_DEADLINE_EXCEEDED` 或链路上最后错误）。
- 可能存在少量 in-flight 子请求尾巴（请求已发出但结果已无业务价值）。
- 不会出现持续重试风暴，**垃圾流量风险低但非零**（短尾）。

---

## 3. timeout = 20ms 行为分析

## 3.1 重试次数

- 结论：与 5ms 类似，**仍接近无重试**。
- 常态表现：大多仍为 1 次尝试。

原因：

- 20ms 依旧显著低于 50ms 的“可继续重试门槛”。

## 3.2 成功路径特征

- 比 5ms 略好，但本质仍是“单次快速尝试”。
- 远程链路成功率依赖极佳网络和负载条件，稳定性通常不足。

## 3.3 timeout 返回与残余流量

- 能快速 timeout 返回。
- 相比大 timeout 配置，尾部流量更小，因为没有足够时间触发多轮重试。

---

## 4. 对业务行为的直接影响

1. **5ms/20ms 配置下，Get 更像“快速失败策略”而不是“重试容错策略”。**
2. **端到端成功率更依赖本地缓存命中率**，而非网络恢复能力。
3. **超时返回的确定性较强**（快速收敛），但 remote get 成功窗口显著缩小。
4. **不会形成大量重试垃圾流量**，仅可能有少量一次性 in-flight 收尾流量。

---

## 5. 建议配置与适用场景

| 超时配置 | 行为类型 | 适用场景 | 风险 |
| --- | --- | --- | --- |
| 5ms | 单次快速失败 | 强本地命中、对尾延迟极端敏感 | 远程成功率极低 |
| 20ms | 近似单次尝试 | 本地优先、可接受少量超时 | 远程链路不稳定 |
| >=80~100ms | 可进入有限重试窗口 | 需要兼顾成功率与时延 | 尾延迟更高 |

建议：

- 若业务目标是“超低延迟 + 明确降级”，可用 5ms/20ms。
- 若业务目标是“尽量成功并可承受一次重试”，建议超时至少跨过 50ms 门槛，实践上推荐 `>=80~100ms`。

---

## 6. 回答核心问题（结论复述）

1. **Get timeout 能否正常返回？**  
   能。当前实现有明确超时预算控制和返回路径。

2. **timeout 返回时是否会有垃圾流量？**  
   可能有少量 in-flight 请求收尾，但不会长时间持续放大；在 5ms/20ms 场景通常更少。
