# 手动验证确认指南

本文供你**亲自逐步执行**，确认 **`yuanrong-datasystem-agent-workbench` + `yuanrong-datasystem`** 分工与脚本可用。请按顺序做；任一步失败时，先看该步的「常见问题」，再查 [`cmake-non-bazel.md`](cmake-non-bazel.md) 与 [`plans/agent开发载体_vibe与yuanrong分工.plan.md`](../../plans/agent开发载体_vibe与yuanrong分工.plan.md)。

**第三方缓存、`output`/`build` 里 Client·Worker·whl·dscli 落哪里**：见 [`构建产物目录与可复现工作流.md`](构建产物目录与可复现工作流.md)。

**本地留档（便于你核对 Agent 的步骤与输出）**：把与下文**完全一致**的命令及标准输出保存到 **`$VIBE/results/`** 下（建议按日期分子目录，并用 `tee`）。该目录下除 [`results/README.md`](../../results/README.md) 外**不入库**；写法与示例见该 README。

---

## 开始前：约定路径

在终端里设两个变量（**改成你本机绝对路径**）：

```bash
export DS="/home/你的用户/workspace/git-repos/yuanrong-datasystem"
export VIBE="/home/你的用户/workspace/git-repos/yuanrong-datasystem-agent-workbench"
export CTEST_OUTPUT_ON_FAILURE=1
```

**目录布局（推荐）**：`$DS` 与 `$VIBE` 的**父目录相同**（例如都在 `git-repos/` 下）。若不同级，必须增加：

```bash
export DATASYSTEM_ROOT="$DS"
```

---

## 步骤 0：确认仓库与文件存在

| 检查项 | 命令 | 成功判据 |
|--------|------|----------|
| datasystem 根 | `test -f "$DS/build.sh" && echo OK` | 输出 `OK` |
| vibe 根 | `test -x "$VIBE/ops" && echo OK` | 输出 `OK` |
| 构建目录（若已编过） | `test -f "$DS/build/CMakeCache.txt" && echo OK` | 输出 `OK`；若没有，步骤 2 需完整构建 |

可选：用 Cursor/VS Code 打开 **`$VIBE/datasystem-dev.code-workspace`**，左侧应出现两个根：`yuanrong-datasystem-agent-workbench` 与 `yuanrong-datasystem`。

---

## 步骤 1：确认 `build.sh` 可用

```bash
cd "$DS"
bash build.sh -h | head -20
```

**成功判据**：打印 Usage，其中可见 `-t`、`-c`、`-p`、`-B` 等选项。

---

## 步骤 2：完成至少一次工程构建（含测试目标）

若 **`$DS/build/CMakeCache.txt` 已存在**且你近期编过，可跳过本步，直接进入步骤 3。

否则在 **`$DS`** 执行（耗时较长，视机器与网络而定）：

```bash
cd "$DS"
bash build.sh -t build -B "$DS/build"
```

**成功判据**：无报错退出；且存在可执行文件：

```bash
test -x "$DS/build/tests/st/ds_st_kv_cache" && echo "ST binary OK"
```

**常见问题**

- 若仅需「能跑 ST」而全量 `build.sh` 太重：可在已配置好的 build 目录内只编测试目标（在 `$DS` 下）：
  ```bash
  cmake --build "$DS/build" --target ds_st_kv_cache -j "$(nproc)"
  ```
  注意：有时仍会触发 CMake/第三方逻辑，属工程行为，非 vibe 脚本问题。

---

## 步骤 3：从 vibe 侧跑 KV executor 门禁（推荐 `--skip-build`）

在**任意当前目录**执行（依赖 `DATASYSTEM_ROOT` 或 `$VIBE` 与 `$DS` 同级）：

```bash
bash "$VIBE/ops" test.kv_executor --skip-build "$DS/build"
```

**成功判据**

1. 第一行类似：`[1/3] Skip build (--skip-build)`  
2. `[2/3]` 后 **CTest** 跑 `KVClientExecutorRuntimeE2ETest`，**8/8 通过**（或与你本地启用的用例数一致，且失败数为 0）  
3. `[3/3]` 后输出 `No forbidden keywords found in src.` 与 `Validation passed.`

**常见问题**

- **`libeSDKOBS.so` 找不到**：不要直接运行 `ds_st_kv_cache` 二进制；必须通过 **本脚本或 `ctest`**。  
- **`Missing or non-executable: .../ds_st_kv_cache`**：先完成步骤 2。  
- **若不用 `--skip-build`**：会执行 `cmake --build ... ds_st_kv_cache`，可能很慢或触发第三方；日常验证优先 `--skip-build`。

---

## 步骤 4：锁竞争 batch perf（单测 + 提示看 PERF 行）

```bash
bash "$VIBE/ops" runtime.lock_perf build
```

注意：参数 **`build`** 是**相对 `$DS` 根**的目录名（脚本会解析为 `$DS/build`），不要求 `cd` 到 `$DS`。

**成功判据**：CTest 中 `PerfConcurrentMCreateMSetMGetExistUnderContention` **Passed**；结尾提示关注 `PERF_CONCURRENT_BATCH`。

**常见问题**

- 耗时约 **1 分钟量级** 属正常。  
- 若从旧路径调用，请优先切换到 **`$VIBE/ops <能力命令>`**。

---

## 步骤 5：URMA/UB IDE 索引库生成

```bash
bash "$VIBE/ops" analysis.refresh_urma_index
```

**成功判据**：终端打印 `Source : .../build/compile_commands.json`，`Output : .../.cursor/compile_commands.json`，`Patched: N`（N>0），无 traceback。

**确认文件**：

```bash
test -f "$DS/.cursor/compile_commands.json" && python3 -c "import json;print(len(json.load(open('$DS/.cursor/compile_commands.json'))))"
```

应输出一个正整数（编译单元条数）。

**常见问题**

- **`Source file not found`**：先在 `$DS` 完成 CMake 配置，确保存在 `build/compile_commands.json`。  
- 若 `$DS` 非同级：先 `export DATASYSTEM_ROOT="$DS"` 再运行脚本。

在 IDE 中打开 **`$DS`** 工程时，若已配置 `.vscode` 使用 `.cursor/compile_commands.json`，可 **Reload Window** 后试跳转 `USE_URMA` 分支。

## 步骤 6（可选）：覆盖率

仅在需要时执行（会拉长构建/测试时间）：

```bash
cd "$DS"
bash build.sh -c on -t build -B "$DS/build"
# 或生成 HTML（示例）
bash build.sh -c html -t run_cpp -l 'st*' -B "$DS/build"
```

**成功判据**：以 `build.sh` 与 CMake 打印为准，无失败退出；HTML 路径看日志。

- [ ] 步骤 2：`ds_st_kv_cache` 可执行（或你接受当前构建状态）  
- [ ] 步骤 3：`validate_kv_executor.sh --skip-build` 全通过 + `Validation passed`  
- [ ] 步骤 4：`run_kv_concurrent_lock_perf.sh` 通过
