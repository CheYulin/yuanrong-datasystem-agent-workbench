# 构建产物目录与可复现工作流（yuanrong-datasystem）

本文说明 **第三方缓存**、**安装产物（Client / Worker / Python·dscli）**、**tests / example 的构建输出**，以及 **CMake 生成物** 的常见位置，并给出一套**简单、可复现**的命令流。路径以仓库根为 **`$DS`**（请替换为本机 `yuanrong-datasystem` 绝对路径）。

更完整的命令组合仍见 [`cmake-non-bazel.md`](cmake-non-bazel.md)；逐步验收见 [`手动验证确认指南.md`](手动验证确认指南.md)。

---

## 1. 第三方（third-party）：路径与避免重复构建

### 1.1 预编译依赖缓存目录 `DS_OPENSOURCE_DIR`

工程通过 CMake（`cmake/util.cmake`）把各依赖的**下载、打补丁、安装前缀**放到同一缓存根目录下，避免每次从零编第三方。

- **环境变量**：`DS_OPENSOURCE_DIR`（与 `build.sh` 帮助中一致）。  
- **未设置时**：默认 **`/tmp/<SHA256>`**，其中哈希来自 **`CMAKE_BINARY_DIR` 的绝对路径**（即与你选的 **`build` 目录**绑定）。

含义：

- **固定使用同一个 `build` 目录**时，第三方缓存路径稳定，重复 `build.sh` 会复用缓存。  
- **若经常换 `build` 路径**（例如今天 `./build`、明天 `./build-debug`），会对应**不同** `/tmp/...` 缓存，看起来像「又在全量编第三方」。

**推荐（多人/多分支可复现、少重复构建）**：显式指定共享缓存，例如：

```bash
export DS_OPENSOURCE_DIR="${HOME}/.cache/yuanrong-datasystem-third-party"
mkdir -p "$DS_OPENSOURCE_DIR"
```

然后在 **`$DS`** 执行 `build.sh`（或 CMake 配置时传入同一变量）。同一台机器上不同 build 目录可共用一份第三方树（注意磁盘占用）。

### 1.2 构建目录内的第三方相关路径

| 路径（相对 `build/`，`$B` 表示你的 CMake build 目录） | 说明 |
|------------------------------------------------------|------|
| **`$B/_deps/`** | CMake `FetchContent`/ExternalProject 等拉取与构建的中间目录（如各库的 `-build`、`-subbuild`）。 |
| **`$B/multi_build/`** | `scripts/build_thirdparty.sh` 并行编第三方时的进度、日志、依赖标记等（与 `build.sh` + 高版本 CMake 流程相关）。 |

**验收**：配置成功后，`cmake` 日志里应有一行类似：`Cache the third party libs to <DS_OPENSOURCE_DIR>`。

## 2. 编译产物路径：安装前缀（Client / Worker / Python·whl·dscli）

`build.sh` 默认 **`CMAKE_INSTALL_PREFIX`** 为 **`$DS/output`**（可用 **`-o`** 改安装目录）。以下均相对于该 **安装根**，记为 **`$OUT`**（默认 `$DS/output`）。

### 2.1 C++ Client SDK（头文件、库、CMake 包）

| 路径 | 内容 |
|------|------|
| **`$OUT/datasystem/sdk/cpp/include/`** | 对外 C++ 头文件 |
| **`$OUT/datasystem/sdk/cpp/lib/`** | 动态/静态库、`cmake/` 下的 **`DatasystemConfig.cmake`** 等，供 `find_package(Datasystem)` |

业务上「**client**」侧链接 SDK 时，通常把 **`CMAKE_PREFIX_PATH`** 指到 **`$OUT/datasystem/sdk/cpp`**（与 `example/cpp/CMakeLists.txt` 用法一致）。

### 2.2 Worker / 服务侧（二进制与配套库）

| 路径 | 内容 |
|------|------|
| **`$OUT/datasystem/service/`** | **`datasystem_worker`** 可执行文件、服务进程依赖的 **`.so`**、配置模板等（具体以安装规则为准） |
| **`$OUT/datasystem/service/lib/`** | 服务侧依赖库集中位置之一（与 `cmake/package.cmake` 中 `DATASYSTEM_SERVICE_*` 一致） |

「**worker**」验收：安装完成后存在可执行的 **`datasystem_worker`**（或你方打包改名后的等价物）。

### 2.3 Python wheel 与 **dscli**

- **`dscli`** 入口声明在 **`setup.py`**（`dscli=yr.datasystem.cli.command:main`），随 **Python 发行包（wheel）** 安装到环境后可用。  
- **`build.sh`** 成功结束后，常见用法是安装 **`$OUT` 下的 `openyuanrong_datasystem-*.whl`**（具体文件名以版本为准，以构建日志 / `pip install` 行为为准）。

wheel **生成阶段**多落在 **build 树**（例如与 `CMAKE_BINARY_DIR` 下的 **`dist/`、`python_lib/`** 等相关），最终是否拷贝到 **`$OUT` 根** 取决于当前版本的打包规则。

**验收命令示例**：

```bash
find "$DS/output" "$DS/build" -maxdepth 5 -name 'openyuanrong_datasystem-*.whl' 2>/dev/null
```

安装后：

```bash
python3 -m pip install --user "$DS/output"/openyuanrong_datasystem-*.whl   # 按实际文件名
command -v dscli && dscli --version
```

## 3. Example 与 Tests 的构建产物路径

### 3.1 Tests（ST/UT 等）

主工程在 **`$B`**（如 **`$DS/build`**）下生成测试可执行文件，例如：

| 路径 | 说明 |
|------|------|
| **`$B/tests/st/ds_st_kv_cache`** | 常见 ST 二进制（名称以 CMake 为准） |
| **`$B/tests/st/*_tests.cmake` 等** | CTest 用 **`cmake -E env`** 包装的运行脚本片段 |

**不要**直接运行上述二进制（易缺 `LD_LIBRARY_PATH`）；请用 **`ctest --test-dir $B`** 或 **`vibe-coding-files/scripts/verify/validate_kv_executor.sh`**。

### 3.2 Example（`build.sh` 触发的 example）

`build.sh` 会在 **`$DS/example/cpp/`** 下创建独立构建目录（历史上多为 **`example/cpp/build`**），在其中对 **已安装 SDK**（默认期望 **`$DS/output/datasystem/sdk/cpp`**）做 `find_package` 并编译示例。

**验收**：示例构建成功后，在 **`$DS/example/cpp/build/`**（或 `build.sh` 日志里打印的路径）下产生示例可执行文件。

## 4. CMake 常见生成物（与 IDE）

| 路径 | 说明 |
|------|------|
| **`$B/CMakeCache.txt`** | 缓存变量、安装前缀、`DS_OPENSOURCE_DIR` 解析结果等 |
| **`$B/compile_commands.json`** | 编译数据库（`CMAKE_EXPORT_COMPILE_COMMANDS`）；URMA 索引脚本读此文件 |
| **`$B/cmake_install.cmake`**、`Makefile` 或 `build.ninja` | 构建系统生成文件 |
| **`$B/src/`** | 部分生成源码、proto 生成物等（以实际目标为准） |

若在 **`$DS`** 根使用软链 **`compile_commands.json` -> build/...**，IDE 打开 **`$DS`** 根即可索引。

## 5. 可复现工作流（建议新人按顺序执行）

以下假设 **`$DS`**、**`$B=$DS/build`**、**`$OUT=$DS/output`**；第三方缓存固定到用户目录，避免换 build 路径时重复拉依赖。

```bash
export DS="/绝对路径/yuanrong-datasystem"
export DS_OPENSOURCE_DIR="${HOME}/.cache/yuanrong-datasystem-third-party"
mkdir -p "$DS_OPENSOURCE_DIR"

cd "$DS"
bash build.sh -B "$B" -o "$OUT"
```

**成功后可核对**：

```bash
test -d "$OUT/datasystem/sdk/cpp/include" && test -d "$OUT/datasystem/sdk/cpp/lib" && echo "C++ SDK layout OK"
test -x "$OUT/datasystem/service/datasystem_worker" && echo "worker binary OK"
find "$OUT" "$B" -maxdepth 6 -name 'openyuanrong_datasystem-*.whl' 2>/dev/null | head -3
test -x "$B/tests/st/ds_st_kv_cache" && echo "ST test binary OK"
```

按需跑测试（仍在 **`$DS`**）：

```bash
export CTEST_OUTPUT_ON_FAILURE=1
ctest --test-dir "$B" -N | head -20    # 先看测试列表
# 或使用 vibe 脚本（见手动验证确认指南）
```

按需跑 example：

```bash
cd "$DS"
bash build.sh -t run_example
```

## 6. 与「仓上组件」的对应关系（便于验收对齐）

| 口头称呼 | 典型落点 |
|----------|-----------|
| **Client（C++）** | **`$OUT/datasystem/sdk/cpp`**（include + lib + CMake 包） |
| **Worker** | **`$OUT/datasystem/service`**（`datasystem_worker` + 服务库） |
| **Python / dscli** | 安装 **`openyuanrong_datasystem-*.whl`** 后，`dscli` 在 `PATH` 中 |
| **Tests** | **`$B/tests/...`** 可执行文件 + CTest 描述 |
| **Example** | **`$DS/example/cpp/build/`**（或 `build.sh` 所用目录） |

若你方内部还有 **「dscli」单独仓库/包**，仍以**当前 CMake 安装与 wheel 命名**为准；上表描述的是本仓库 **`build.sh` + CMake** 的默认布局。
