# 数据系统可观测性设计（重构版）

## 1. 设计目标

本文档面向 KVC SDK、KVC Worker、URMA 协同链路，目标是建立一套可落地的可观测性体系，支撑以下能力：

1. **快速定界**：在成功率下降、时延劣化时，快速判断问题属于 SDK、Worker、网络链路、URMA 还是外部依赖。
2. **精准定位**：基于 Trace 串联请求全链路日志，定位具体失败阶段与错误原因。
3. **低开销观测**：通过请求采样、异步刷盘和日志滚动压缩，降低对业务读写路径影响。
4. **可运营治理**：通过指标与阈值告警，形成“发现问题 -> 定界 -> 定位 -> 处置”的闭环。

---

## 2. 系统范围与部署上下文

### 2.1 关键组件

- **KVC Worker**：每节点每业务 1 个（参考 618 规模：64 节点）。
- **推理实例（集成 KVC SDK）**：每节点 8 个（主图:子图 = 3:1，约每节点 2 个子图实例）。
- **etcd 集群**：5 副本（1 主 + 4 备），独立部署。
- **二级存储（分布式网盘）**：全局共享，Worker 通过 POSIX 接口访问。
- **监控与日志平台**：
  - 思南（Grafana）：指标可视化。
  - DongMonitor：Worker 资源监控。
  - 行云平台：SDK/Worker 日志采集与检索。

### 2.2 请求链路（简化）

1. SDK 接收读写请求。
2. 本地命中走共享内存，异常或跨节点场景走 TCP/UB。
3. Worker 完成元数据查询、远端数据拉取和传输。
4. etcd 提供租约保活与故障检测，二级存储承担持久化与恢复。

## 3. 可观测性总体设计

可观测信号分三层：

1. **日志（Log）**：还原请求上下文与错误细节，用于精准定位。
2. **指标（Metrics）**：刻画成功率、时延和流量，用于快速定界和趋势判断。
3. **告警（Alert）**：基于阈值触发，保障异常可被及时发现。

三层通过 `TraceID` 打通：监控先发现异常，随后按 Trace 反查 SDK/Worker/URMA 运行日志定位根因。

---

## 4. 日志方案

### 4.1 日志分类

1. **运行日志**：关键路径阶段日志、状态变更日志。
2. **错误日志**：系统调用失败、UMDK/URMA 接口失败、参数非法、内部错误等。
3. **接口日志（access）**：接口名、成功状态、时延、数据量等，用于监控平台统计。

### 4.2 日志 Header 规范

统一 Header 字段建议包含：

- 时间戳
- 日志级别
- 文件名与行号
- Pod/实例标识
- `PID:TID`
- `TraceID`
- 集群名/业务标识

### 4.3 请求采样与 Trace 关联

- **请求粒度采样**：支持可配置采样率（如 1%~100%）打印关键路径日志，控制日志规模。
- **全链路 Trace**：SDK、Worker、URMA 侧日志共享同一 `TraceID`，支持端到端检索。
- **URMA 协同**：数据系统向 URMA 注册日志回调，在 URMA 日志中注入同源 Trace 信息。

### 4.4 性能与可靠性策略

- **异步刷盘**：业务线程写内存缓冲，后台线程异步落盘，避免阻塞主流程。
- **磁盘写满/队列满保护**：触发自动丢弃策略，优先保障核心业务可用性。
- **异步滚动与压缩**：定期轮转与压缩日志，控制单文件大小与存储成本。

## 5. 指标方案（Metrics）

### 5.1 采集周期与输出

- 默认按 **10s 粒度**周期统计并输出（可配置）。
- 重点采集：
  - 成功率
  - P50/P99 时延（直方图）
  - 请求/传输数据量（计数器）

### 5.2 关键路径指标

| 组件 | 流程阶段 | 指标类型 | 指标示例 |
|---|---|---|---|
| SDK | 读取请求-共享内存 | P99 时延 | `client_get_local_latency` |
| SDK | 读取请求-UB 写 | P99 时延 | `client_get_urma_write_latency` |
| SDK | 读取请求-TCP 写 | 数据量 | `client_get_tcp_write_total_bytes` |
| Worker | 元数据查询-TCP | P99 时延 | `worker_rpc_query_meta_latency` |
| Worker | 远端拉取请求-TCP | P99 时延 | `worker_rpc_get_remote_latency` |
| Worker | 远端传输-UB | P99 时延 | `worker_worker_urma_write_latency` |
| Worker | 远端传输-TCP | 数据量 | `worker_worker_tcp_write_total_bytes` |

### 5.3 建议补齐指标（UB 建链）

| 组件 | 流程阶段 | 指标类型 | 指标示例 |
|---|---|---|---|
| SDK | 读取请求-UB 建链 | P99 时延 | `client_worker_urma_import_jfr_latency` |
| SDK | 读取请求-UB 建链 | 次数 | `client_worker_urma_import_jfr_count` |
| Worker | 远端传输-UB 建链 | P99 时延 | `worker_worker_urma_import_jfr_latency` |
| Worker | 远端传输-UB 建链 | 次数 | `worker_worker_urma_import_jfr_count` |

### 5.4 指标判因原则

1. **先看结果指标**：成功率 / 全链路 P99 是否异常。
2. **再看阶段指标**：对比子流程 P99，识别劣化阶段。
3. **最后看传输流量**：结合 UB/TCP bytes 增量判断是否降级、拥塞或流量异常。

## 6. 告警方案

### 6.1 接口与链路告警（10s 窗口）

- 接口 P99 延迟持续 > **2ms**。
- 接口成功率 < **99%**（或较基线下降 > 1 个百分点）。
- URMA 传输成功率 < **99%**。
- RPC 成功率 < **99%**。
- etcd 请求成功率 < **99%**。
- 二级存储请求成功率 < **99%**。

### 6.2 资源与容器告警

- Worker 共享内存使用率持续 > **90%**。
- Pod CPU 使用率持续 10s >= **100%**（按容器配额口径）。
- Pod 内存使用率 > 资源上限的 **90%**。
- Liveness/Readiness 连续失败 >= **3 次**。

### 6.3 告警落地要求

- 告警必须携带：集群、实例、组件、接口/阶段、阈值、观测窗口、最近 `TraceID` 示例。
- 告警消息直达运维与值班群，支持一键跳转监控面板与日志检索。

## 7. 定界与定位 Runbook

### 7.1 定界输入

- 推理实例请求成功率
- 端到端 P99 时延
- 容器健康状态与资源指标

### 7.2 排障步骤

1. **查 SDK 接口监控**：确认是否为入口请求异常（成功率/时延）。
2. **查 Worker 资源与接口监控**：判断是否资源瓶颈、线程池堆积、访问失败。
3. **查三件依赖（etcd、二级存储、网络链路）**：确认外部依赖是否异常。
4. **按 Trace 拉通日志**：回放单请求全链路，定位失败阶段和错误码。

### 7.3 常见故障归类

1. UB 链路故障
2. TCP 链路故障
3. Worker 资源异常或进程异常
4. etcd/二级存储访问失败

## 8. 错误码分层与处置建议

### 8.1 错误分层

1. **客户集成错误**：参数非法、Key 不合法、接口使用顺序错误等。
2. **部署初始化错误**：Worker 未就绪、初始化连接失败等。
3. **内部处理错误**：运行时错误、内存不足等。
4. **RPC 错误**：网络故障、超时、服务不可达。
5. **URMA 错误**：UB 传输异常、链路建连异常等。

### 8.2 典型错误码（示意）

| 错误码 | 含义 | 常见原因 | 处置建议 |
|---|---|---|---|
| `K_DUPLICATED` | Key duplicated | 重复创建已存在 Key | 修正调用逻辑 |
| `K_INVALID` | Invalid key | Key 格式/长度非法 | 修正参数并加校验 |
| `K_NOT_FOUND` | Key not found | 读取不存在对象 | 校验业务时序或兜底 |
| `K_NOT_READY` | Worker not ready | Worker 未就绪/不可用 | 重试 + 检查 Worker 状态 |
| `K_OUT_OF_MEMORY` | Out of memory | Worker 共享内存不足 | 降流、扩容、清理策略 |
| `K_RPC_UNAVAILABLE` | RPC unavailable | 网络故障/超时/内部错误 | 排查网络与超时配置 |
| `K_URMA_ERROR` | URMA transmission error | UB 传输或建链异常 | 按 URMA 指标与日志定位 |
| `K_RUNTIME_ERROR` | Runtime error | 数据系统内部错误 | 按 Trace 检索日志定位 |

### 8.3 故障处置建议（统一口径）

- **客户集成错误**：由客户侧修正业务逻辑和参数校验。
- **初始化错误**：重试并关注实例重启后恢复情况。
- **RPC/URMA/内部错误**：若持续 5s 以上，先降流；降流后仍持续则执行切流。

---

## 9. 观测数据量与采样建议

- 参考跨节点 Get 示例，请求级日志量可达多条（SDK + Worker + access）。
- 建议按业务等级配置采样：
  - 稳态：低采样（如 1%~5%）
  - 异常窗口：临时升采样（如 20%~100%）
- 采样策略需支持动态生效，避免频繁发版调整。

---

## 10. 验收标准（建议）

1. 任一告警触发后，值班同学可在 **5 分钟内完成故障定界**。
2. 任一请求可通过 `TraceID` 关联到 SDK、Worker、URMA 关键日志。
3. 日志系统在磁盘压力下不阻塞核心请求路径。
4. 关键路径 P99 指标与流量指标完整、可视化、可告警。

以上标准用于验证“可观测性体系”已可用于生产值守与故障处置。
